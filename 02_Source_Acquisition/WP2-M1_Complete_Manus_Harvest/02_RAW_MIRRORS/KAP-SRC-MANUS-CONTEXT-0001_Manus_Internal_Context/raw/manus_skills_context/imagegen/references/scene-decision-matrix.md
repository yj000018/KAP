# Scene Decision Matrix

Use this reference when a visual request could be satisfied in multiple ways. Choose the scenario whose failure mode would make the result most unusable.

## Decision Rule

Identify the image's **job** before describing its look. The same subject requires different decisions depending on medium. A coffee cup for a marketplace needs product accuracy; a coffee cup for a website hero needs layout support; a coffee cup for a sticker needs transparent edges and simple silhouette.

## Scenario Matrix

| Scenario | Primary job | Prioritize | Avoid | Default assumptions when unspecified |
| --- | --- | --- | --- | --- |
| Website hero | Support landing page message and conversion | Wide composition, subject on one side, copy-safe whitespace, restrained background, brand-compatible palette | Busy detail behind text, centered subject blocking headline, high contrast noise, awkward crop on mobile | Use 16:9 or wide landscape; leave 35–50% negative space for copy |
| Website section visual | Explain a feature or concept | Clear metaphor, modular composition, consistent style with page | Overly cinematic visuals that distract from UI/content | Use clean 3D/vector/editorial style unless user requests realism |
| Blog or article cover | Attract attention and summarize topic | Strong conceptual image, readable focal point, safe crop | Literal clutter, small text, dense labels | Use 16:9 or 3:2 with central safe area |
| Product hero | Present product desirably and accurately | Accurate shape, material, perspective, lighting, clean background | Deformed product, invented labels, inconsistent scale, wrong packaging | Use studio lighting, simple background, commercial photography or 3D render |
| Marketplace product image | Meet listing clarity | Full product visible, neutral background, minimal props | Lifestyle clutter, cropped edges, altered brand marks | Use square format with centered subject unless user says otherwise |
| Packaging concept | Explore visual direction | Front-facing readable structure, color system, shelf impact | Too much unreadable microtext, impossible dielines | Treat as concept art, not print-ready production artwork |
| Social ad or campaign image | Stop scrolling and communicate emotion | Bold focal point, high contrast hierarchy, platform-safe crop | Too many elements, tiny text, weak call-to-action area | Use platform ratio if provided; otherwise offer one main aspect ratio and note assumption |
| Poster | Create a composed key visual with title, subtitle, callouts, or event copy when needed | Strong hierarchy, readable typography, dramatic layout, balanced margins, accurate requested text | Missing required message, overcrowding, weak focal point, blank-background generation followed by Python/scripted text overlay | Directly generate the finished poster with specified text in the requested language. | Thumbnail | Create instant recognition at small size | Large subject, simple background, high contrast, expressive emotion, readable short title if requested | Small details, low-contrast palettes, cluttered microtext | Use 16:9, one primary subject, and directly render a short title or label when it improves communication |
| UI mockup | Show a credible product interface as an image | Information architecture, consistent components, realistic spacing, readable labels, believable device or browser context | Nonsensical controls, fake dense text, inconsistent button styles, over-engineered implementation workflows | Directly generate the UI image as a visual mockup unless the authoritative routing in `SKILL.md` indicates code, static layout, or functional implementation |
| Dashboard visual | Convey data-product feel as a visual image | Clean grid, readable cards, believable charts, consistent spacing, concise labels and numbers | Chart junk, impossible numbers, chaotic widgets, HTML screenshot detours for a visual-only request | Directly generate the dashboard image with plausible or provided data labels; use code only for functional dashboards, exact chart data, or editable implementation requests |
| App icon | Be memorable at small sizes | Simple silhouette, high contrast, centered symbol | Tiny detail, long text, complex scenes | Use square format; create symbol-first concepts |
| Logo concept | Explore brand mark direction | Simplicity, scalability, distinct shape, brand fit | Precise typography dependence, gradients that fail in mono, copied-looking marks | Generate symbol-only or wordless concepts unless exact wordmark requested |
| Icon set | Build consistent UI assets | Shared stroke weight, perspective, corner radius, fill style | Mixed styles, inconsistent proportions, too much detail | Use vector-like simple icons on transparent or plain background |
| Game prop | Produce reusable asset | Clear silhouette, consistent camera angle, clean edges, style matching | Cropped subject, background contamination, perspective mismatch | Ask or infer 2D/3D, isometric/side/top-down, transparent background need |
| Game character sprite | Fit game view and animation needs | Pose readability, silhouette, consistent proportions, limited detail | Cinematic portrait instead of asset, inconsistent view angle | Use neutral pose or requested action; transparent background by default if asset-like |
| Game background | Support gameplay or scene mood | Depth layers, readable play area, perspective consistency | Overly busy foreground, unclear walkable space | Match requested camera view; leave functional space where needed |
| Character portrait | Express identity and mood | Face consistency, natural anatomy, outfit anchors, lighting | Extra fingers, changed identity, inconsistent age, style drift | Define identity anchors before generating variations |
| Mascot | Create repeatable character system | Strong silhouette, simple shapes, brand colors, expressive pose | Too many accessories, hard-to-repeat details | Use clean character design with clear front view first |
| Transparent cutout | Create compositable subject | Alpha transparency, complete object, clean edge, no fringe | White box, colored halo, unwanted shadow, clipped subject | Request transparent background and enough padding around subject |
| Sticker | Be fun and reusable | Bold outline, simple readable shape, transparent background | Thin edges, photoreal clutter, text errors | Use centered subject, thick outline, transparent background unless otherwise stated |
| Image edit | Modify existing image | Local change accuracy, preservation of untouched areas | Global style changes, identity drift, lighting mismatch | State exactly what to change and what to preserve |
| Image upscale or restore | Enhance resolution and restore details of an existing image while preserving all original content | Exact preservation of original content, improved clarity, higher resolution | Hallucinated new elements, style drift, loss of original identity, added or removed objects | Use the image variation tool with the exact upscale/restore prompt from `prompt-recipes.md`; do not add extra instructions or rephrase it |
| Background replacement | Change environment while keeping subject | Subject boundary, lighting consistency, perspective match | Halo edges, wrong shadows, changed subject | Preserve subject identity, pose, clothing, product geometry |
| Text replacement or localization | Replace or translate visible image text | Exact target text, same placement and style when requested, preserved surrounding image | Missing target text, changed layout, unrelated visual changes | Use image editing directly first; reserve design/layout post-processing only for explicit editable source, pixel-perfect brand typography, or legally exact microtext |
| Infographic | Explain accurate information as a finished visual | Valid structure, readable labels, clean arrows, logical grouping, accurate requested copy | Fabricated facts, unreadable text, visual overload, background-only generation followed by Python/scripted text overlay | First organize the content into title, sections, labels, callouts, and hierarchy; then directly generate the complete infographic with embedded text. Do not use background-plus-overlay workflows because the text is Chinese, non-English, multilingual, or dense; use deterministic layout only when exact data, formal structure, editable source, legal microtext, or machine-readable requirements explicitly require it |
| Structured diagram | Clarify exact systems, flows, dependencies, sequences, or hierarchies | Structural accuracy, consistent connectors, readable labels, correct sequence | Decorative ambiguity, invented nodes, wrong sequence, unreadable labels, AI-generated graph art replacing a formal diagram | Use Mermaid or deterministic rendering first when nodes, edges, sequence, hierarchy, or machine-readable structure must be accurate. Use AI generation only for a separate polished conceptual version after the structure is clear |

## Choosing Between Beauty and Correctness

When a tradeoff appears, prefer correctness over visual flourish, but do not turn correctness into open-ended self-auditing. A less dramatic product image that preserves the product is better than a beautiful but inaccurate one. A generated UI mockup, infographic, poster, or menu with requested labels rendered clearly is preferable to an unnecessary multi-step workflow. For formal structured diagrams, correctness usually means Mermaid or deterministic rendering first; for conceptual explanatory visuals, correctness may mean a clear generated composition. Do not externalize text merely because it is Chinese, non-English, multilingual, or lengthy; organize the copy first, then directly generate the finished image. A blank/background-only image plus Python/scripted text overlay is forbidden as a workaround for text rendering. Use deterministic layout only when the user explicitly requires exact typography, legal fine print, exact long tables, machine-readable structure, editable source, or print-production files.

Once a generated visual is good enough for the user's explicit request, deliver it and stop. Do not keep cropping local regions, previewing labels, searching fonts, writing repair scripts, expanding masks, or generating extra variants merely to increase confidence. Those actions are appropriate only when the user asked for them or a visible critical defect makes the delivered artifact unusable.

## When To Ask The User

Ask a concise question only when the missing answer changes the final image. Important questions include final medium, required aspect ratio, exact text/copy, required language, brand colors, reference style, product constraints, transparent background need, or whether a source image must be preserved.

Do not ask for model parameters or implementation details unless the user explicitly requests them.
