# ELYSIUM: Style Guide

## 1. Voice and Tone
- **Serious & Architectonic**: Write like a systems architect diagnosing a complex structure.
- **Diagnostic**: Focus on what is structurally wrong and how it functions, not just symptoms.
- **Systems-Oriented**: Emphasize flows, feedback loops, and metabolic processes.
- **Literate but Clear**: Use precise, evocative language, but avoid academic opacity.
- **Philosophical but Operational**: Ground abstract concepts in actionable, structural realities.
- **Civilizational Scale**: Maintain a macro perspective.

**Avoid:**
- Academic opacity and jargon.
- Activist sloganism or moralizing.
- Vague, ungrounded spirituality.
- Techno-utopian hype.

## 2. Structural Rhythm
Each major section should follow this rhythmic progression:
1. **Strong diagnostic opening**: State the structural reality immediately.
2. **Concrete systemic example**: Ground the diagnosis in reality.
3. **Structural principle**: Extract the underlying rule or failure mode.
4. **Architectural implication**: What does this mean for the whole?
5. **Transition**: Bridge smoothly to the next section.

## 3. Prose Rules
- **Avoid excessive repetition**: Trust the reader to remember established concepts.
- **Preserve conceptual recurrence**: Re-use core terms (metabolism, architecture, extraction) when conceptually necessary to build the ontology.
- **Use "architecture" consistently**: Treat it as a diagnostic term for how things are built and connected.
- **Precision over variety**: Avoid overusing "systemic" when "structural," "metabolic," or "architectural" is more precise.
- **Holistic connection**: Each chapter must explicitly connect back to the whole ELYSIUM ontology.

## 4. Markdown Standards
- **H1**: One `#` per module (the module title).
- **H2**: `##` for major sections.
- **H3**: `###` for subsections.
- **Emphasis**: Use `**bold**` only for key claims, definitions, or core concepts. Do not over-bold.
- **Tables**: Use Markdown tables for architectural breakdowns, matrices, and comparisons. Do not use tables for dense prose.
- **Lists**: Use bullet points for diagnostic categories, failure modes, and matrix steps.
- **Spacing**: Preserve readable blank lines between paragraphs and sections.
